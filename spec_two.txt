PROJECT EXTENSION: Client System Capability Detection & Dynamic File Size Limiter

OBJECTIVE:
Enhance the offline encryption web app to detect the user’s current system capabilities (memory, CPU hints, browser limits) and dynamically restrict the maximum allowed file size to prevent crashes during encryption/decryption.

This must run entirely in the browser and work offline.

CORE REQUIREMENTS:

1. SYSTEM SPEC DETECTION (BROWSER-SAFE)
   Collect and display the following (no fingerprinting for tracking; display only, do not transmit):

* navigator.hardwareConcurrency → logical CPU cores
* navigator.deviceMemory (if available) → RAM in GB (approximate)
* userAgent (parsed to show browser name only)
* platform (Windows/Linux/macOS/Android/iOS)
* Web Crypto availability check
* Web Worker availability check
* IndexedDB availability check

Display in UI panel:
“System Capability”

Example output:
CPU Cores: 8
Approx RAM: 8 GB
Browser: Chrome
WebCrypto: Available
WebWorkers: Available

2. MEMORY-SAFE MAX FILE SIZE CALCULATION

Define safe usable memory:

If navigator.deviceMemory exists:
usableMemoryBytes = deviceMemoryGB * 1024^3 * 0.25
Else:
assume 512MB usable

Set:
maxFileSize = usableMemoryBytes * 0.5

Hard caps:

* Minimum max file size: 25 MB
* Maximum allowed: 1 GB

Rationale:
Encryption requires multiple buffers (plaintext, ciphertext, IV, salt, overhead). Limiting to 50% of safe memory prevents tab crashes.

3. DYNAMIC UI LIMIT

Display:

“Maximum safe file size for this device: XXX MB”

When user selects a file larger than allowed:

* Block operation
* Show error:
  “File exceeds safe limit for your device. Please use a smaller file.”

4. STREAM MODE (OPTIONAL ENHANCEMENT)

If file > 50MB AND browser supports ReadableStream:

* Enable chunked encryption mode
* Chunk size: 1 MB
* Show progress bar

If streams unsupported:

* Fallback to full buffer mode
* Still enforce maxFileSize

5. WEB WORKER MEMORY SAFETY

Crypto operations must run in Web Worker.

Use:

* Transferable ArrayBuffers
* Release references after each chunk
* Explicitly overwrite buffers with zeros after use

6. PERFORMANCE CLASSIFICATION

Based on deviceMemory and hardwareConcurrency:

LOW TIER:

* deviceMemory <= 2GB OR cores <= 2
* Max file size capped at 100MB
* Force chunk mode

MID TIER:

* deviceMemory 4–8GB
* Max file size up to 500MB

HIGH TIER:

* deviceMemory >= 8GB AND cores >= 6
* Allow up to 1GB

Display:
“Performance Tier: Low / Medium / High”

7. SECURITY CONSTRAINTS

* Do not store system specs
* Do not send system specs over network
* Do not use specs as encryption key material
* Specs are for UI + limits only

8. TEST CASES

Simulate and validate:

* Low memory device (2GB) → limit enforced at ~100MB
* Mid device (4GB) → limit ~500MB
* High device (8GB+) → limit ~1GB
* File just below limit → succeeds
* File just above limit → blocked with message

9. DELIVERABLES

Implement:

/lib/system.ts

* getSystemCapabilities()
* calculateMaxFileSize()

/components/SystemPanel.tsx

* Displays system info and tier

Update file picker logic to:

* Check file.size against dynamic limit before encryption

10. SUCCESS CRITERIA

* App never crashes due to large file selection
* Limit adapts per device
* Works offline
* No sensitive data leakage
* Accurate UI reporting

END OF SPEC
